# Requirements Document

## Introduction

This document defines requirements for six new core features in the GROWTHOVO React Native / Expo self-improvement app that form the "Daily OS" — the daily operating system that drives engagement, accountability, and real behaviour change:

1. **Morning Briefing** — A mandatory full-screen daily briefing card triggered at a user-set time, containing a mental state check-in, Rex's daily truth, a single focus action, a streak/league snapshot, and accountability partner status.
2. **SOS Button** — An always-visible emergency button on the home screen that surfaces six situation-specific intervention flows (anxiety, reactive anger, zero motivation, hard conversation, habit urge, overwhelm).
3. **Rex Memory System + Always-On Chat** — A persistent memory layer that stores goals, struggles, wins, patterns, promises, and people, combined with a 24/7 bottom-sheet chat interface where Rex references real memory in every conversation.
4. **Evening Debrief** — A mandatory three-question AI-powered evening reflection triggered at a user-set time, feeding Rex's memory and generating a closing verdict with XP reward.
5. **Accountability Partner** — A paired accountability system where two users see each other's check-in status, receive partner notifications, and get a weekly comparison report.
6. **Weekly Rex Report** — A full cinematic Sunday-night premium report with AI-generated pattern analysis, a Rex voice verdict delivered via OpenAI TTS, and a shareable card.

All features use TypeScript, dark-mode-first design, GROWTHOVO brand colours, react-native-reanimated animations, Supabase with RLS on every new table, and i18next for multilingual support (EN, RO, IT, FR, DE, ES, PT, NL).

## Glossary

- **Morning_Briefing**: The mandatory full-screen card shown before the home screen each morning at the user's configured time.
- **Briefing_Service**: The TypeScript service at `src/services/briefingService.ts` responsible for generating and persisting morning briefing data.
- **Mental_State**: One of five emoji-coded states selected by the user during the morning check-in: Anxious, Low, Neutral, Good, Locked_In.
- **Daily_Focus**: A single, specific, time-bound action generated by Rex based on the user's weakest pillar of the week.
- **SOS_Button**: The always-visible emergency button on the home screen that opens the SOS bottom sheet.
- **SOS_Service**: The TypeScript service at `src/services/sosService.ts` responsible for logging SOS events and generating Rex responses.
- **SOS_Type**: One of six situation categories: `anxiety_spike`, `about_to_react`, `zero_motivation`, `hard_conversation`, `habit_urge`, `overwhelmed`.
- **Rex_Memory**: A structured record in the `rex_memory` Supabase table capturing a single piece of information about the user.
- **Memory_Type**: One of six categories for Rex_Memory entries: `goal`, `struggle`, `win`, `pattern`, `promise`, `person`.
- **Rex_Chat_Service**: The TypeScript service at `src/services/rexChatService.ts` responsible for the always-on chat interface.
- **Evening_Debrief**: The mandatory three-question evening reflection triggered at the user's configured time.
- **Debrief_Service**: The TypeScript service at `src/services/debriefService.ts` responsible for persisting debrief data and triggering Rex analysis.
- **Accountability_Pair**: A bidirectional pairing between two users stored in the `accountability_pairs` Supabase table.
- **Partner_Message**: A short motivational message sent from one accountability partner to another, stored in `partner_messages`.
- **Weekly_Rex_Report**: The full cinematic Sunday-night premium report stored in `weekly_rex_reports`.
- **Report_Service**: The TypeScript service at `src/services/reportService.ts` responsible for fetching and displaying the weekly report.
- **TTS_Audio**: An audio file generated by the OpenAI TTS API (model: `tts-1-hd`, voice: `onyx`) and stored in Supabase Storage.
- **Pillar**: One of six skill categories: Mind, Discipline, Communication, Money, Career, Relationships.
- **Premium_User**: A user whose `subscription_status` is `active` or `trialing`.
- **Free_User**: A user whose `subscription_status` is `free` or `canceled`.
- **Rex**: The AI coach persona powered by OpenAI, with a defined system prompt enforced on every call.
- **RLS**: Row Level Security enforced on all Supabase Postgres tables.

## Requirements

### Requirement 1: Morning Briefing — Trigger and Display

**User Story:** As a user, I want a mandatory morning briefing card to appear before the home screen each day, so that I start every day with intention and accountability.

#### Acceptance Criteria

1. THE Briefing_Service SHALL schedule a daily push notification at the user's configured morning time (default 07:30 local time) with the message "Rex has your briefing. 60 seconds."
2. WHEN the user opens the app after the morning notification, THE App SHALL display the Morning_Briefing full-screen card before navigating to the home screen.
3. WHILE the Morning_Briefing card is displayed, THE App SHALL prevent the user from dismissing it until all five sections have been viewed and the mental state has been selected.
4. WHEN the user taps the "Let's go" dismiss button, THE Briefing_Service SHALL record `briefing_viewed_at` in the `daily_checkins` table and award 10 XP to the user.
5. THE Briefing_Service SHALL display the Morning_Briefing at most once per calendar day per user.
6. THE App SHALL complete the full Morning_Briefing interaction in under 60 seconds for a user who reads at average pace.
7. IF the Briefing_Service fails to load AI-generated content within 5 seconds, THEN THE Briefing_Service SHALL display pre-written fallback content for Rex's Daily Truth and Today's Single Focus without blocking the briefing.

### Requirement 2: Morning Briefing — Mental State Check-In

**User Story:** As a user, I want to tap one emoji to log how I feel each morning, so that Rex can personalise his response and track my emotional patterns over time.

#### Acceptance Criteria

1. THE Morning_Briefing SHALL display a single mental state question with exactly five emoji options: 😰 Anxious, 😔 Low, 😐 Neutral, 🙂 Good, 🔥 Locked_In.
2. WHEN the user taps a mental state emoji, THE Briefing_Service SHALL immediately display a one-sentence Rex reaction specific to that state.
3. WHEN the user taps a mental state emoji, THE Briefing_Service SHALL store the selected state and Rex's reaction in `daily_checkins(user_id, date, morning_state, morning_rex_response)`.
4. THE Briefing_Service SHALL use the model `gpt-4o-mini` to generate the Rex reaction, with a maximum of 20 tokens.
5. IF the OpenAI call for the Rex reaction exceeds 3 seconds, THEN THE Briefing_Service SHALL display a pre-written fallback reaction for that mental state without surfacing an error to the user.
6. THE Briefing_Service SHALL store exactly one mental state record per user per calendar day; subsequent taps on the same day SHALL update the existing record.

### Requirement 3: Morning Briefing — Rex's Daily Truth

**User Story:** As a user, I want Rex to deliver a personalised daily truth each morning, so that I receive a push that is specific to my situation rather than generic motivation.

#### Acceptance Criteria

1. THE Briefing_Service SHALL generate Rex's Daily Truth using the model `gpt-4o-mini` with a maximum of 2 sentences and 60 tokens.
2. WHEN generating Rex's Daily Truth, THE Briefing_Service SHALL include in the prompt: the user's primary pillar, the last 3 evening debrief obstacle answers, the current streak count, and the day of week.
3. WHEN the day of week is Monday, THE Briefing_Service SHALL instruct the model to deliver a harder push in Rex's Daily Truth.
4. WHEN the day of week is Friday, THE Briefing_Service SHALL instruct the model to deliver a reflection prompt in Rex's Daily Truth.
5. THE Briefing_Service SHALL never generate a Rex's Daily Truth that contains generic motivational phrases such as "You've got this", "Believe in yourself", or "Every day is a new opportunity".
6. IF the OpenAI call for Rex's Daily Truth fails or exceeds 5 seconds, THEN THE Briefing_Service SHALL display a pre-written fallback truth from a pool of at least 14 entries (2 per day of week).

### Requirement 4: Morning Briefing — Today's Single Focus

**User Story:** As a user, I want Rex to give me one specific real-world action to do today, so that I have a concrete task tied to my weakest area rather than a vague goal.

#### Acceptance Criteria

1. THE Briefing_Service SHALL generate Today's Single Focus as exactly one action item, not a lesson, that is specific and time-bound.
2. WHEN generating Today's Single Focus, THE Briefing_Service SHALL base it on the user's weakest pillar of the current week, determined by the lowest XP earned across pillars in the last 7 days.
3. THE Briefing_Service SHALL use the model `gpt-4o-mini` with a maximum of 30 tokens for Today's Single Focus.
4. THE generated focus action SHALL include a time constraint (e.g., "Before 2pm:", "Tonight:") and a specific observable behaviour.
5. IF the OpenAI call for Today's Single Focus fails or exceeds 5 seconds, THEN THE Briefing_Service SHALL display a pre-written fallback focus action from a pool of at least 6 entries per pillar.

### Requirement 5: Morning Briefing — Streak, League, and Partner Snapshot

**User Story:** As a user, I want to see my streak, hearts, league position, and accountability partner status in my morning briefing, so that I have full context on my standing before the day begins.

#### Acceptance Criteria

1. THE Morning_Briefing SHALL display the user's current streak count, remaining hearts, and current league position.
2. WHEN the user's league position has dropped since the previous day, THE Briefing_Service SHALL include a Rex comment on the drop.
3. WHEN the user's streak is at a milestone (7, 14, 30, 60, or 90 days), THE Briefing_Service SHALL display a special milestone message instead of the standard streak display.
4. WHEN the user has an active Accountability_Pair, THE Morning_Briefing SHALL display the partner's check-in status: either "[Name] checked in. They're ready." or "[Name] hasn't checked in yet. You going to let them slip?"
5. WHEN the user does not have an active Accountability_Pair, THE Morning_Briefing SHALL display a prompt to invite an accountability partner.
6. THE Briefing_Service SHALL fetch streak, hearts, and league data from Supabase with a maximum query time of 3 seconds before falling back to locally cached values.

### Requirement 6: SOS Button — Visibility and Entry

**User Story:** As a user in a moment of crisis, I want a large, always-visible SOS button on the home screen, so that I can access help in one tap without searching through menus.

#### Acceptance Criteria

1. THE App SHALL display the SOS_Button on the home screen at all times, not hidden behind menus, tabs, or scroll positions.
2. WHEN the user taps the SOS_Button, THE App SHALL present a bottom sheet within 300 milliseconds displaying six situation options.
3. THE SOS bottom sheet SHALL display exactly six options with their labels: 😰 Anxiety Spike, 😤 About to React Badly, 😔 Zero Motivation, 😬 Hard Conversation Coming, 🔥 About to Break a Habit, 🌀 Overwhelmed / Too Much.
4. WHEN the user selects a situation, THE SOS_Service SHALL log an `sos_events` record with `user_id`, `type`, `timestamp`, and `outcome` set to `started`.
5. THE SOS_Service SHALL update the `outcome` field in `sos_events` to `completed` or `abandoned` when the user finishes or exits the flow.
6. THE SOS_Service SHALL record `duration_seconds` in `sos_events` as the elapsed time from flow start to completion or abandonment.

### Requirement 7: SOS — Anxiety Spike Flow

**User Story:** As a user experiencing an anxiety spike, I want a guided breathing exercise followed by a cognitive reframe, so that I can interrupt the anxiety response and regain control.

#### Acceptance Criteria

1. WHEN the user selects Anxiety Spike, THE App SHALL display an animated 4-7-8 breathing exercise with visual guidance for inhale (4 seconds), hold (7 seconds), and exhale (8 seconds).
2. THE breathing exercise animation SHALL use react-native-reanimated and complete a minimum of 3 full cycles before the user can proceed.
3. AFTER the breathing exercise, THE App SHALL display the cognitive reframe prompt: "Name 3 things actually in your control right now."
4. WHEN the user submits the cognitive reframe, THE SOS_Service SHALL display a Rex closing line generated by `gpt-4o-mini` that references the user's anxiety history (frequency of anxiety SOS events in the last 30 days).
5. THE SOS_Service SHALL use a maximum of 25 tokens for the Rex closing line in the Anxiety Spike flow.
6. IF the user has triggered the Anxiety Spike SOS 3 or more times in the current week, THEN THE SOS_Service SHALL flag this pattern and include it in the next evening debrief Rex analysis.

### Requirement 8: SOS — About to React Badly Flow

**User Story:** As a user about to send an angry message or react impulsively, I want a forced pause followed by help drafting a calm response, so that I avoid saying something I'll regret.

#### Acceptance Criteria

1. WHEN the user selects About to React Badly, THE App SHALL display the message "Put the phone down for 90 seconds first." with an animated 90-second countdown timer.
2. THE 90-second timer SHALL use a calming visual animation (e.g., expanding circle) and SHALL NOT allow the user to skip it.
3. AFTER the 90-second timer completes, THE App SHALL display the prompt "Still want to send it? Here's what to say instead."
4. WHEN the user confirms they still want to proceed, THE SOS_Service SHALL call `gpt-4o-mini` to help the user draft a calm response, accepting a 2–3 word situation description from the user as input.
5. THE SOS_Service SHALL use a maximum of 80 tokens for the calm response draft.
6. IF the OpenAI call for the calm response draft exceeds 5 seconds, THEN THE SOS_Service SHALL display a pre-written fallback calm response template.

### Requirement 9: SOS — Zero Motivation Flow

**User Story:** As a user with zero motivation, I want a 60-second Rex reset that references my actual goals, so that I get a personalised push rather than generic inspiration.

#### Acceptance Criteria

1. WHEN the user selects Zero Motivation, THE SOS_Service SHALL call `gpt-4o-mini` to generate a 60-second reset message referencing the user's stated goals from Rex_Memory and their current streak.
2. THE reset message SHALL end with exactly one micro-action that is physical and immediately executable (e.g., "Do one push-up. Right now. Then decide if you still feel the same.").
3. THE SOS_Service SHALL use a maximum of 100 tokens for the Zero Motivation reset message.
4. IF the user has no Rex_Memory entries of type `goal`, THEN THE SOS_Service SHALL generate the reset message based on the user's primary pillar instead.
5. IF the OpenAI call exceeds 5 seconds, THEN THE SOS_Service SHALL display a pre-written fallback reset message from a pool of at least 5 entries.

### Requirement 10: SOS — Hard Conversation Coming Flow

**User Story:** As a user facing a difficult conversation, I want Rex to give me an opening line, things to avoid, and a target outcome, so that I go in prepared rather than reactive.

#### Acceptance Criteria

1. WHEN the user selects Hard Conversation Coming, THE App SHALL display a text input asking "What's the conversation?" with a placeholder of "2–3 words, e.g. 'firing someone', 'asking for raise'".
2. WHEN the user submits the situation description, THE SOS_Service SHALL call `gpt-4o-mini` to generate: an opening line, a list of things to avoid saying, and a target outcome to aim for.
3. THE SOS_Service SHALL use a maximum of 150 tokens for the Hard Conversation preparation response.
4. WHEN the user is a Premium_User, THE App SHALL display a "Rehearse with Rex" option after the preparation response.
5. WHEN the user taps "Rehearse with Rex", THE SOS_Service SHALL enter a multi-turn chat mode where Rex plays the other person in the conversation and the user practices via text.
6. THE "Rehearse with Rex" feature SHALL be available to Premium_Users only; Free_Users SHALL see a paywall prompt instead.
7. THE SOS_Service SHALL complete the Hard Conversation preparation (without rehearsal) in under 3 minutes of total interaction time.

### Requirement 11: SOS — About to Break a Habit Flow

**User Story:** As a user about to break a habit, I want a guided urge-surfing experience that helps me ride out the craving, so that I don't act on impulse and can log a success.

#### Acceptance Criteria

1. WHEN the user selects About to Break a Habit, THE App SHALL display a list of the user's tracked habits for selection.
2. WHEN the user selects a habit, THE SOS_Service SHALL display the urge surfing message: "The urge will pass in 4–7 minutes. Don't feed it. Watch it like a wave."
3. WHEN the user is a Premium_User, THE SOS_Service SHALL play a 5-minute guided audio track generated by the OpenAI TTS API (model: `tts-1-hd`, voice: `onyx`) for urge surfing.
4. WHEN the user is a Free_User, THE SOS_Service SHALL display a text-based urge surfing guide instead of audio.
5. THE SOS_Service SHALL display a distraction challenge — a specific physical action to perform immediately — after the urge surfing content.
6. WHEN the user returns to the app after 10 or more minutes following the urge surfing flow, THE SOS_Service SHALL update the `sos_events` record `outcome` to `success` and award 15 XP.
7. IF the urge surfing audio generation fails, THEN THE SOS_Service SHALL fall back to the text-based guide without surfacing an error.

### Requirement 12: SOS — Overwhelmed / Too Much Flow

**User Story:** As a user feeling overwhelmed, I want to brain-dump everything in my head and have Rex sort it into what matters today versus what can wait, so that I can regain focus.

#### Acceptance Criteria

1. WHEN the user selects Overwhelmed / Too Much, THE App SHALL display a free-text input (or voice note option) for the user to dump everything in their head.
2. THE brain dump input SHALL accept a minimum of 10 words before the user can submit.
3. WHEN the user submits the brain dump, THE SOS_Service SHALL call `gpt-4o-mini` to return: the 1 thing that actually matters today, the 3 things to ignore until tomorrow, and one sentence to reset with.
4. THE SOS_Service SHALL use a maximum of 120 tokens for the Overwhelmed response.
5. IF the OpenAI call exceeds 5 seconds, THEN THE SOS_Service SHALL display a pre-written fallback structure: "Focus on one thing. Ignore the rest until tomorrow. Breathe."
6. THE SOS_Service SHALL store the brain dump text in the `notes` field of the `sos_events` record.

### Requirement 13: Rex Memory System — Storage and Structure

**User Story:** As a user, I want Rex to remember what I've told him across sessions, so that every interaction feels personal and builds on what came before rather than starting from scratch.

#### Acceptance Criteria

1. THE Rex_Chat_Service SHALL store Rex_Memory entries in the `rex_memory` Supabase table with columns: `id`, `user_id`, `memory_type`, `content`, `importance_score`, `created_at`, `last_referenced_at`.
2. THE `memory_type` field SHALL accept exactly six values: `goal`, `struggle`, `win`, `pattern`, `promise`, `person`.
3. THE `importance_score` field SHALL be an integer between 1 and 10 inclusive.
4. THE Rex_Chat_Service SHALL extract and store Rex_Memory entries from the following sources: evening debrief answers, morning state check-ins, SOS interactions, explicit chat with Rex, and challenge completions and failures.
5. WHEN a Rex_Memory entry is referenced in a Rex response, THE Rex_Chat_Service SHALL update `last_referenced_at` to the current timestamp.
6. THE Rex_Chat_Service SHALL enforce RLS on `rex_memory` so that users can only read and write their own memory entries.
7. THE Rex_Chat_Service SHALL cap the total number of Rex_Memory entries per user at 200; WHEN the cap is reached, THE Rex_Chat_Service SHALL delete the entry with the lowest `importance_score` and oldest `last_referenced_at` before inserting a new one.

### Requirement 14: Rex Memory System — Memory Extraction

**User Story:** As a system, I want Rex to automatically extract meaningful memories from user interactions, so that the memory bank grows organically without requiring manual input from the user.

#### Acceptance Criteria

1. WHEN an evening debrief is submitted, THE Debrief_Service SHALL call `gpt-4o-mini` to extract up to 3 Rex_Memory entries from the Q2 obstacle answer and Q3 note, assigning appropriate `memory_type` and `importance_score` values.
2. WHEN a morning mental state of Anxious or Low is selected 3 or more times in a 7-day window, THE Briefing_Service SHALL create a Rex_Memory entry of type `pattern` with `importance_score` 8.
3. WHEN an SOS event of type `anxiety_spike` occurs 3 or more times in a 7-day window, THE SOS_Service SHALL create a Rex_Memory entry of type `pattern` with `importance_score` 9.
4. WHEN the user explicitly tells Rex something in chat (e.g., "I want to...", "My goal is...", "I'm struggling with..."), THE Rex_Chat_Service SHALL extract and store a Rex_Memory entry of the appropriate type.
5. THE memory extraction prompt SHALL use a maximum of 200 tokens and SHALL NOT store trivial or duplicate information.

### Requirement 15: Rex Always-On Chat — Interface

**User Story:** As a user, I want to chat with Rex at any time from the home screen, so that I can get personalised coaching whenever I need it without navigating to a separate screen.

#### Acceptance Criteria

1. THE App SHALL display a Rex chat entry point on the home screen that is accessible at all times.
2. WHEN the user taps the Rex chat entry point, THE App SHALL slide up a bottom sheet chat interface within 300 milliseconds.
3. THE Rex chat bottom sheet SHALL feel like a messaging interface, not a full-screen app — it SHALL display a scrollable message history and a text input at the bottom.
4. WHEN Rex opens a new conversation, THE Rex_Chat_Service SHALL include at least one reference to a Rex_Memory entry in Rex's opening message.
5. THE Rex_Chat_Service SHALL use the model `gpt-4o-mini` for all chat responses with a maximum of 3 sentences (approximately 80 tokens) unless the user explicitly asks for more detail.
6. THE Rex_Chat_Service SHALL include the Rex system prompt (as defined in the design document) in every OpenAI API call.
7. THE Rex_Chat_Service SHALL include up to 5 of the most relevant Rex_Memory entries (by `importance_score` descending, then `last_referenced_at` descending) in the context of every OpenAI API call.
8. THE Rex_Chat_Service SHALL never generate responses containing: "Great question!", "Amazing!", "I understand how you feel", corporate language, or generic advice that ignores the user's memory context.
9. WHEN the user closes the chat bottom sheet, THE Rex_Chat_Service SHALL persist the conversation history to Supabase so it is available when the chat is reopened.
10. THE Rex_Chat_Service SHALL be available to all users (Free and Premium); Premium_Users SHALL receive AI-generated responses while Free_Users SHALL receive fallback responses from a pre-written pool.

### Requirement 16: Evening Debrief — Trigger and Structure

**User Story:** As a user, I want a mandatory three-question evening debrief triggered at my set time, so that I reflect on my day and give Rex the data he needs to coach me better.

#### Acceptance Criteria

1. THE Debrief_Service SHALL schedule a daily push notification at the user's configured evening time (default 21:00 local time) with the message "Rex is waiting. 2 minutes."
2. WHEN the user opens the app after the evening notification, THE App SHALL display the Evening_Debrief full-screen flow before navigating to the home screen.
3. THE Evening_Debrief SHALL present exactly three questions, one at a time, in order: Q1 (daily focus completion), Q2 (obstacles), Q3 (note for tomorrow).
4. THE Debrief_Service SHALL display the Evening_Debrief at most once per calendar day per user.
5. WHEN the user completes the Evening_Debrief, THE Debrief_Service SHALL award 20 XP and update the user's streak.
6. WHEN the user completes the Evening_Debrief, THE Debrief_Service SHALL store all answers in `evening_debriefs(id, user_id, date, q1_answer, q1_detail, q2_obstacle, q3_note, rex_verdict, xp_earned)`.
7. WHEN the user completes the Evening_Debrief, THE Debrief_Service SHALL display a preview of tomorrow's focus.

### Requirement 17: Evening Debrief — Question Flow and Enforcement

**User Story:** As a user, I want the debrief questions to require honest answers before I can proceed, so that I can't tap through without actually reflecting.

#### Acceptance Criteria

1. WHEN Q1 is displayed, THE App SHALL show three response options: "Yes, crushed it", "Partially", "No, didn't happen".
2. WHEN the user selects "Yes, crushed it" for Q1, THE App SHALL display a follow-up free-text prompt: "What made it work?"
3. WHEN the user selects "Partially" for Q1, THE App SHALL display a follow-up free-text prompt: "What got in the way?"
4. WHEN the user selects "No, didn't happen" for Q1, THE App SHALL display Rex's prompt "What actually happened?" and SHALL NOT allow the user to proceed to Q2 until they have entered at least 10 words.
5. WHEN Q2 is displayed, THE App SHALL show a free-text or voice note input with the label "What tried to stop you today?" and SHALL enforce a minimum of 10 words before the user can proceed.
6. WHEN the user submits Q2, THE Debrief_Service SHALL call `gpt-4o-mini` to generate one insight from the obstacle answer and display it immediately.
7. WHEN Q3 is displayed, THE App SHALL show a free-text input with the label "What does tomorrow-you need to know?" — this question is optional but encouraged.
8. WHEN Q3 is submitted, THE Debrief_Service SHALL store the note and incorporate it into the next morning's briefing generation prompt.
9. AFTER all three questions, THE Debrief_Service SHALL call `gpt-4o-mini` to generate a 2-sentence closing verdict using the full debrief context and store it in `rex_verdict`.

### Requirement 18: Evening Debrief — Weekly Pattern Analysis

**User Story:** As a user, I want Rex to analyse my last 7 debriefs every Sunday, so that I receive a pattern report that reveals what's actually holding me back.

#### Acceptance Criteria

1. WHEN Sunday arrives, THE Debrief_Service SHALL trigger a weekly pattern analysis for all users who have submitted at least 3 evening debriefs in the past 7 days.
2. THE weekly pattern analysis SHALL use the model `gpt-4o` and include all Q2 obstacle answers and Q3 notes from the past 7 days as context.
3. THE weekly pattern analysis output SHALL be stored and surfaced as input to the Weekly_Rex_Report generation.
4. IF a user has fewer than 3 evening debriefs in the past 7 days, THEN THE Debrief_Service SHALL skip the weekly pattern analysis for that user.

### Requirement 19: Accountability Partner — Setup and Pairing

**User Story:** As a user, I want to invite one accountability partner via SMS or WhatsApp, so that someone who knows me is watching my progress and I feel real social pressure to show up.

#### Acceptance Criteria

1. WHEN a user initiates accountability partner setup, THE App SHALL generate a shareable invite message: "I'm using GROWTHOVO to work on [pillar]. I need you to hold me accountable. Download the app: [link]".
2. THE App SHALL allow the user to send the invite via the native share sheet (SMS, WhatsApp, or any installed app).
3. WHEN the invited person downloads the app and accepts the pairing, THE App SHALL create an `accountability_pairs` record with `user_id`, `partner_id`, `pillar`, `active: true`, and `created_at`.
4. THE invited partner SHALL receive full accountability features (check-in status visibility, partner notifications, weekly comparison) without requiring a Premium subscription.
5. THE App SHALL allow each user to have at most one active Accountability_Pair at a time.
6. WHEN a user deactivates an Accountability_Pair, THE App SHALL set `active` to `false` in the `accountability_pairs` record.
7. THE App SHALL display the accountability partner setup prompt during onboarding and in the profile screen.

### Requirement 20: Accountability Partner — Daily Mechanics

**User Story:** As a user, I want to see my partner's check-in status and receive notifications about their activity, so that we create mutual accountability pressure throughout the day.

#### Acceptance Criteria

1. THE App SHALL display the accountability partner's morning check-in status on the home screen: either a green indicator ("checked in") or a grey indicator ("not checked in yet").
2. WHEN a user completes their morning briefing, THE App SHALL send a push notification to their partner: "[Name] checked in. They're going for day [X]."
3. WHEN a user has not submitted their evening debrief by 22:00 local time, THE App SHALL send a push notification to their partner: "[Name] hasn't checked in tonight. Send them a message?" with quick reply options: "You got this 💪", "Don't break it 🔥", "I'm watching 👀".
4. WHEN a partner sends a quick reply message, THE App SHALL store it in `partner_messages(id, pair_id, sender_id, message, message_type, created_at)` and display it as a Rex-style card on the recipient's home screen.
5. WHEN a user breaks their streak, THE App SHALL notify their partner before any other notification: "[Name] just broke their streak. Send them a comeback message."
6. WHEN a partner sends a comeback message after a streak break, THE App SHALL display it prominently on the relapse screen.
7. THE App SHALL use Supabase Realtime to update partner check-in status on the home screen without requiring a manual refresh.

### Requirement 21: Accountability Partner — Weekly Comparison Report

**User Story:** As a user, I want a weekly comparison with my accountability partner, so that friendly rivalry keeps both of us engaged and competing to improve.

#### Acceptance Criteria

1. WHEN Sunday at 20:00 local time arrives, THE App SHALL generate a weekly comparison report for each active Accountability_Pair.
2. THE weekly comparison report SHALL include: challenges completed by each partner, current streak of each partner, SOS events used by each partner, and a winner declaration.
3. THE App SHALL send a push notification to both partners with the comparison result: "This week: [Name] won. Keep it up." or "This week: it was close. Don't let them win next week."
4. THE weekly comparison report SHALL be displayed as a card on the home screen for both partners.
5. THE weekly comparison report SHALL use friendly competitive framing and SHALL NOT shame the losing partner.

### Requirement 22: Weekly Rex Report — Trigger and Structure

**User Story:** As a premium user, I want a full cinematic weekly report every Sunday evening, so that I get a comprehensive honest assessment of my week with Rex's voice reading the verdict.

#### Acceptance Criteria

1. THE Report_Service SHALL generate the Weekly_Rex_Report for Premium_Users only, every Sunday at 20:00 local time.
2. THE App SHALL send a push notification at 20:00 on Sunday: "Your week is done. I have thoughts."
3. THE Weekly_Rex_Report SHALL consist of four sections in order: The Numbers, Rex's Pattern Analysis, The Verdict, Next Week's Focus.
4. WHEN the Weekly_Rex_Report is generated, THE Report_Service SHALL store it in `weekly_rex_reports(id, user_id, week_start, numbers_json, pattern_analysis, verdict_text, audio_url, next_week_focus_json, created_at)`.
5. IF a Weekly_Rex_Report already exists for a given `user_id` and `week_start`, THEN THE Report_Service SHALL return the cached record without regenerating it.
6. THE Report_Service SHALL enforce RLS on `weekly_rex_reports` so users can only read their own reports.

### Requirement 23: Weekly Rex Report — Content Generation

**User Story:** As a premium user, I want Rex's pattern analysis and verdict to be specific to my actual week's data, so that the report feels like a real coach's assessment rather than a template.

#### Acceptance Criteria

1. THE Numbers section SHALL auto-calculate and display: lessons completed, challenges done vs missed, SOS moments by type, morning check-in streak, evening debrief completion rate, and XP earned.
2. WHEN generating Rex's Pattern Analysis, THE Report_Service SHALL use the model `gpt-4o` with the full week of debrief data, SOS logs, and morning check-in states as context.
3. THE Pattern Analysis SHALL output exactly 3 specific observations about the user's behaviour patterns, each in 1–2 sentences.
4. WHEN generating The Verdict, THE Report_Service SHALL use the model `gpt-4o` to produce one honest, specific paragraph (maximum 150 tokens) with no generic phrases.
5. THE Next Week's Focus section SHALL specify: one pillar to focus on, one habit to build, and one challenge to do differently — all generated by `gpt-4o` based on the pattern analysis.
6. IF the `gpt-4o` API call fails or exceeds 10 seconds, THEN THE Report_Service SHALL use pre-written fallback content for Pattern Analysis and Verdict from a pool of at least 5 entries.

### Requirement 24: Weekly Rex Report — Voice Verdict and Share

**User Story:** As a premium user, I want to hear Rex read the verdict aloud in his voice, so that the weekly report feels cinematic and emotionally impactful rather than just text on a screen.

#### Acceptance Criteria

1. WHEN The Verdict is generated, THE Report_Service SHALL call the OpenAI TTS API (model: `tts-1-hd`, voice: `onyx`) to generate an audio file of Rex reading the verdict.
2. THE generated audio file SHALL be stored in Supabase Storage at path `weekly_reports/{userId}/{week_start}/verdict.mp3` and the URL stored in `audio_url`.
3. WHEN the user reaches The Verdict section of the report, THE App SHALL auto-play the audio file.
4. THE App SHALL display a play/pause control for the verdict audio.
5. IF the TTS API call fails, THEN THE Report_Service SHALL display the verdict text only, without audio, and log the failure.
6. THE App SHALL display a share button that generates a shareable card containing: Rex's verdict as a quote, the user's top stat of the week, and GROWTHOVO branding.
7. THE shareable card SHALL include a pre-written caption: "Rex just read me in 3 sentences. @growthovo #growthovo".
8. WHEN the user taps share, THE App SHALL invoke the native share sheet with the card image and pre-written caption.

### Requirement 25: Database Schema — New Tables

**User Story:** As a developer, I want all new Supabase tables created via migration files with RLS enabled, so that the schema is version-controlled, secure, and reproducible.

#### Acceptance Criteria

1. THE migration SHALL create the `daily_checkins` table with columns: `id`, `user_id`, `date`, `morning_state`, `morning_rex_response`, `briefing_viewed_at`; with RLS allowing users to read and write only their own rows.
2. THE migration SHALL create the `evening_debriefs` table with columns: `id`, `user_id`, `date`, `q1_answer`, `q1_detail`, `q2_obstacle`, `q3_note`, `rex_verdict`, `xp_earned`; with RLS allowing users to read and write only their own rows.
3. THE migration SHALL create the `sos_events` table with columns: `id`, `user_id`, `type`, `timestamp`, `duration_seconds`, `outcome`, `notes`; with RLS allowing users to read and write only their own rows.
4. THE migration SHALL create the `rex_memory` table with columns: `id`, `user_id`, `memory_type`, `content`, `importance_score`, `created_at`, `last_referenced_at`; with RLS allowing users to read and write only their own rows.
5. THE migration SHALL create the `accountability_pairs` table with columns: `id`, `user_id`, `partner_id`, `pillar`, `active`, `created_at`; with RLS allowing both `user_id` and `partner_id` to read the row.
6. THE migration SHALL create the `partner_messages` table with columns: `id`, `pair_id`, `sender_id`, `message`, `message_type`, `created_at`; with RLS allowing both members of the pair to read messages.
7. THE migration SHALL create the `weekly_rex_reports` table with columns: `id`, `user_id`, `week_start`, `numbers_json`, `pattern_analysis`, `verdict_text`, `audio_url`, `next_week_focus_json`, `created_at`; with RLS allowing users to read only their own rows.
8. WHEN migrations are applied, THE App SHALL enforce a UNIQUE constraint on `(user_id, date)` for `daily_checkins` and `evening_debriefs`, and a UNIQUE constraint on `(user_id, week_start)` for `weekly_rex_reports`.
